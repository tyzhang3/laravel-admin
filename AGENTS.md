# AGENTS.md

本文件用于指导在本仓库内协作的智能体（Agent）行为规范。

## 1. 核心原则

- 总是用简单直接的思路进行编程，避免过度设计。
- 先分析需求，再拆解任务，再执行修改。
- 优先交付可运行、可验证的结果。
- 防御性编程保持审慎：除非必要，不堆砌冗余判断。
- `try-catch` 可按需使用，其余防御逻辑必须有明确收益。

## 2. 需求处理流程

每次开始任务时，按以下顺序执行：

1. 明确目标：一句话复述用户要什么。
2. 拆解任务：列出关键子任务和依赖关系。
3. 先给建议：说明推荐做法、范围和取舍。
4. 再实施：直接改代码或文档，不只停留在方案。
5. 做验证：至少运行相关测试或最小验证命令。
6. 结果汇报：说明改了什么、为什么、如何验证。

## 3. 编码与修改约束

- 变更应小步、聚焦，避免一次性大面积重构。
- 优先复用现有代码和项目约定，不引入不必要新模式。
- 注释只写必要信息，避免解释显而易见的语句。
- 不主动引入破坏性命令（如重置未请求的改动）。
- 发现与任务无关的异常改动时，先暂停并确认再继续。

## 4. 沟通风格

- 使用简洁、直接、可执行的表达。
- 对复杂任务先给“下一步建议”，再给实施细节。
- 汇报中应包含：
  - 修改文件路径
  - 关键行为变化
  - 验证方式与结果
  - 风险和后续建议（如有）

## 5. 默认交付标准

- 能跑：改动后功能可执行。
- 能测：有最小可行验证。
- 能读：结构清晰，命名明确。
- 能审：变更范围可追踪、可解释。
